## Два варианта архитектуры

Первый из них - монолит. Он проще в разработке, быстрее, так как меньше от сети зависит, но его труднее масштабировать и расширять.
Второй вариант - микросервисная архитектура. Она более распределенная и задачи, которые она решает, более гранулированы. Ее легче масштабировать, в  нашей ситуации это критично, так как например каталог будет нагружем больше оплаты. Она может разрабатываться несколькими командами. Но для всего этого нужна инфра.

С учетом всего этого, преимущественно фактора масштабирования, я выбрал микроархитектурный подход

## Домены

1. *Catalog*
- Ответственность: управление товарами продавцов
- Данные: товары, категории, остатки

2. *Order*
- Ответственность: создание и отслеживание заказов
- Данные: заказы, статусы, состав заказа

3. *Cart*
- Ответственность: управление корзиной
- Данные: товары в корзине пользователя

4. *Payment*
- Ответственность: обработка платежей
- Данные: транзакции, статусы оплаты

5. *Recommendations*
- Ответственность: персональные рекомендации
- Данные: история покупок, ML модели

6. *Notifications*
- Ответственность: отправка уведомлений
- Данные: шаблоны, история отправки

## Сервисы

- *Catalog Service* - Catalog Domain
- *Order Service* - Order Domain
- *Cart Service* - Cart Domain
- *Payment Service* - Payment Domain
- *Recommendation Service* - Recommendations Domain
- *Notification Service* - Notifications Domain

Разбиенеи независимо

## Запуск сервиса

### Сборка образа
```
docker build -t service .
```

### Запуск контейнера
```bash
docker run -d -p 8080:8080 service
```

### Проверка health

http://localhost:8080/health


### Остановка сервиса
```
docker stop service
docker rm service
```
